## Purpose & When To Read This

Open this doc whenever the design system needs raster or texture assets: hero imagery,
illustration sets, background textures, grain/halftone treatments, WebGL textures, normal
maps, or matcaps — whether generated by AI, photographed, or 3D-rendered. It owns the
art-direction brief, prompt-consistency mechanics, asset treatments, web export specs
(formats, resolutions, color profiles), and the IP/licensing risk register. It does NOT own
color-token architecture (see [05-color](./05-color.md#specifications--parameters)) or the
build-time texture compression pipeline (KTX2 etc. — see
[09-tech-implementation](./09-tech-implementation.md#specifications--parameters)).

## Core Principles

1. **Art direction before generation.** Never prompt ad hoc. Every asset traces back to a
   written brief (template in §Specifications) that pins medium, lighting, palette tokens,
   and a consistency mechanism. One stunning rogue image is worth less than ten coherent ones.
2. **Consistency is engineered, not hoped for.** Lock seeds, style references, and reference
   images; record them in a generation log. A model's default style drifts between versions
   (Midjourney's default switched V7 → V8.1 on 2026-06-10/11) — always pin the model version.
3. **Treatment unifies sources.** A shared post-treatment (duotone, grain, halftone) makes
   generated, photographic, and rendered assets read as one system. Apply treatments in the
   pipeline or at render time, never baked inconsistently per asset.
4. **Generate large, ship small.** Generate at ≥ 2× final display size at the model's native
   maximum (FLUX.2: 4 MP; Nano Banana Pro: 4K; Firefly Image 5: 4 MP native), then downscale
   and compress to AVIF/WebP under the per-slot byte budgets in §Specifications.
5. **Legal hygiene is a deliverable.** Purely AI-generated output is not copyrightable in the
   US (USCO report, 2025-01-29). Choose tools and document human authorship deliberately;
   keep a generation log for disclosure and reproducibility.
6. **Numbers, not vibes.** Every export carries exact dimensions, quality settings, color
   profile, and a byte ceiling. "Looks fine" is not a spec.

## Decision Framework

**1. Source selection — generated vs photographic vs 3D-rendered:**

- IF the asset depicts a real, identifiable product, person, team, or place → **photography**
  (or a 3D product render from CAD). AI faces/products read as fake and create likeness risk.
- IF the brand must OWN the copyright in the hero artwork (exclusivity, enforcement) →
  **human-authored** photo/illustration/3D. Purely AI-generated work has no US copyright;
  competitors may legally reuse it.
- IF the asset will be re-lit, re-framed per breakpoint, animated, or needs physically exact
  reflections/transparency → **3D render** (repeatable camera, consistent materials; pairs
  with the WebGL pipeline — see [08-webgl-effects](./08-webgl-effects.md#decision-framework)).
- IF the asset is abstract, conceptual, editorial-illustrative, or a texture/background, and
  budget/timeline excludes a shoot → **AI generation** under the brief + consistency kit.
- IF risk-averse client (regulated industry, litigation-shy) → AI only via **Adobe Firefly**
  (licensed training data + IP indemnification on paid/business plans) or CC0 sources.
- ALWAYS allowed: AI for moodboards, comps, and ideation that humans then re-author.

**2. Tool routing (verified models, 2026-06-12):**

- Cohesive stylized art direction, fast exploration → Midjourney V8.1 (default since
  2026-06-10/11; 4 s SD / 12 s HD generations). Use V7 when `--oref` is required.
- Photoreal + multi-image reference consistency (up to 10 refs) → FLUX.2 (Pro/Flex API).
- Precise instruction-following edits, text-in-image, transparent PNG sprites →
  gpt-image-1.5 / gpt-image-2 (gpt-image-2 does NOT support transparent backgrounds —
  use gpt-image-1/1.5 for cutouts).
- Native 4K, multilingual in-image text, up to 14 input refs → Nano Banana Pro
  (Gemini 3 Pro Image). Note SynthID watermark; visible watermark only off on Ultra tier.
- Indemnified/commercially safe, layered editing, custom brand models → Firefly Image 5.
- Vector/SVG output, locked brand style sets → Recraft V3 (native SVG paths, custom styles
  from uploaded references).

**3. Upscale vs regenerate:**

- IF source ≥ 50% of target pixel dimensions → upscale once, max 2×, fidelity-mode upscaler
  (Topaz Gigapixel; free: Real-ESRGAN/Upscayl). Never chain upscales.
- IF source < 50% of target, or artifacts visible at 100% zoom → regenerate at higher native
  resolution instead of upscaling.
- Generative upscalers (Magnific, Topaz "creative" modes) hallucinate detail — acceptable
  for abstract/texture work; FORBIDDEN on faces, products, logos, UI screenshots.

**4. Licensing gate (run before any asset ships):** see risk register in §Specifications.
   Any "in the style of [living artist / named stock library]" prompt → reject. Any visible
   watermark-like artifact in output → reject (trademark exposure per Getty v Stability AI).

## Specifications & Parameters

### 4.1 Art-direction brief template (fill before generating anything)

```markdown
# Art-Direction Brief — <project>
- archetype: <ref to 01-visual-motion archetype>            # see ./01-visual-motion.md
- medium: <e.g. 35mm film photo | risograph | 3D clay render | gouache illustration>
- era/reference axis: <e.g. 1970s NASA documentation | Swiss 1959 print>
- lighting: <e.g. single hard key 45° left, deep shadows | overcast softbox>
- lens/camera (photo-style only): <e.g. 85mm f/1.8, eye level | macro 100mm>
- palette anchors: <3–5 token refs, e.g. {color.brand.600}, {color.bg.primary}>  # tokens owned by 05-color
- texture treatment: <duotone | grain 8% | halftone 6px/45° | none>  # specs in §4.3
- composition rules: <subject offset to right third; ≥30% negative space for type>
- banned elements (negative list): <text, watermarks, logos, extra fingers, lens flare, ...>
- consistency kit: <model+version, seed(s), --sref/refs URLs, custom style ID>
- licensing tier: <indemnified-only | open | unrestricted> + human-authorship plan
- output matrix: <slot × aspect ratio × master resolution>   # ratios from §4.2 table
```

### 4.2 Prompt engineering for visual consistency

**Prompt skeleton (all models):**
`[subject] + [framing/composition] + [lighting] + [palette words matching tokens] + [medium/style] + [params]`.
Keep a fixed "style suffix" string (medium + lighting + palette + treatment words) reused
verbatim across every asset in a set; vary only the subject clause.

**Consistency mechanisms by model (verified 2026-06-12):**

| Mechanism | Midjourney (V8.1 / V7) | FLUX.2 | gpt-image-1.5/2 | Nano Banana Pro |
|---|---|---|---|---|
| Seed | `--seed 0–4294967295` | seed param in API | n/a (use refs) | n/a (use refs) |
| Style reference | `--sref <url(s)|code>`, weight `--sw 0–1000` (default 100); `--sref random` to explore; pre-2025-06 codes need `--sv 4` | up to 10 reference images per request | image input + edit instructions | up to 14 input images; holds likeness of up to 5 people |
| Subject/character ref | `--oref <url>` + `--ow 0–1000` (default 100; keep ≤ 400) — **V7 only**; V8.1 successor in development | multi-reference (same 10-image budget) | image-to-image edit | multi-image blending |
| Negative prompt | `--no <item, item>` | in prompt ("no …") | in prompt | in prompt |

- Current Midjourney default `--sv` value: UNVERIFIED — confirm before use (check `/settings`).
- Workflow: lock ONE accepted image as the style anchor → feed it as `--sref`/reference to
  every subsequent generation → re-anchor only by explicit art-direction decision.
- Batch discipline: generate candidate sets of 12–20, cull to the 3–5 that sit inside the
  brief, promote 1 to style anchor.

**Aspect ratios mapped to web slots** (generate native to ratio — never crop a 1:1 to 21:9):

| Slot | Ratio | Master (generate/retain) | Ship (max srcset width) |
|---|---|---|---|
| Desktop full-bleed hero | 16:9 | 3840×2160 | 2560w |
| Ultrawide/cinematic hero | 21:9 | 3840×1646 | 2560w |
| Mobile hero (separate art) | 4:5 | 2160×2700 | 1080w |
| Editorial/portrait card | 2:3 or 3:4 | 2048×3072 / 2048×2732 | 960w |
| Card / thumbnail | 4:3 or 1:1 | 1600×1200 / 1600×1600 | 800w |
| Open Graph / social | 1.91:1 | 2400×1260 | ship exactly 1200×630 JPEG |

gpt-image-2 size constraints: any size with max edge ≤ 3840 px, both edges multiples of 16,
aspect ≤ 3:1, total pixels 655,360–8,294,400. Nano Banana Pro tiers: 1K/2K/4K (up to
4096×4096); published ratio list incl. 1:1, 2:3, 3:2, 3:4, 4:3, 4:5, 5:4, 9:16, 16:9, 21:9
(enumeration from third-party API guides: UNVERIFIED — confirm before use).

### 4.3 Asset-treatment specs

| Treatment | Spec | Notes |
|---|---|---|
| Duotone | Desaturate (Rec. 709 luma) → 2-stop gradient map: shadows = dark brand token, highlights = light token; keep OKLCH lightness delta ≥ 0.55 between stops | Token values from [05-color](./05-color.md#specifications--parameters); SVG recipe in §Code Examples |
| Film grain | `feTurbulence type="fractalNoise" baseFrequency 0.65–0.9, numOctaves 2–3`; overlay opacity 0.04–0.12; blend `overlay` or `soft-light` | Or pre-baked 256×256 tiling grain PNG ≤ 8 KB, `background-repeat`; animate at 8–12 fps via `steps()` only with reduced-motion fallback ([07](./07-animation-choreography.md#specifications--parameters)) |
| Noise/dither for gradients | 1–2% monochrome noise over large soft gradients | Kills AVIF/WebP banding; apply BEFORE encoding |
| Halftone | Dot pitch 4–10 px; screen angles 45° (mono) or 15°/45°/75° (multi-ink look); dot size driven by luma | Static: SVG `<pattern>`; animated/hover: shader — catalog owned by [08-webgl-effects](./08-webgl-effects.md#specifications--parameters) |
| Displacement-ready export | 8-bit grayscale PNG, mid-gray 128 = neutral; Gaussian blur σ 2–4 px (prevents stepping); power-of-two: 512/1024/2048; ≤ 256 KB | Runtime displacement usage: [08-webgl-effects](./08-webgl-effects.md#specifications--parameters) |

Treatment order (fixed): retouch → upscale/downscale to master → treatment (duotone/halftone)
→ grain/noise → encode. Grain applied after encoding-scale resize so grain size is consistent
on screen across breakpoints.

### 4.4 Web export specs

**Formats & quality (encode from the lossless master, never from a JPEG):**

| Format | Setting | Use |
|---|---|---|
| AVIF (primary) | avifenc: `--min 0 --max 63 -a end-usage=q -a cq-level=18 -a tune=ssim`, speed 6; alpha: `--minalpha 0 --maxalpha 63`; cq-level range 0–63, lower = higher quality (web.dev) | All photographic/illustrative content; >50% savings vs JPEG is common |
| WebP (fallback) | Lossy quality 75–85 (below 75 artifacts appear; above 85 bytes grow faster than quality) | `<source>` fallback; animated-GIF replacement |
| JPEG (last fallback) | Quality 80, progressive, 4:2:0 chroma | `<img src>` fallback; OG images (crawlers: JPEG/PNG only) |
| PNG | Only for hard-edged alpha cutouts ≤ 1000 px and the lossless masters | Otherwise use AVIF/WebP with alpha |

**Per-slot byte budgets (at largest shipped width):** hero AVIF ≤ 200 KB (target 80–150 KB);
card ≤ 50 KB; thumbnail ≤ 20 KB; texture/grain overlays ≤ 30 KB total per page; OG JPEG
≤ 300 KB. Page-level image budgets and LCP/CLS targets are owned by
[09-tech-implementation](./09-tech-implementation.md#specifications--parameters).

**Responsive ladder:** widths 640 / 960 / 1280 / 1600 / 1920 / 2560 — these are delivery
sizes (rendered CSS width × DPR, up to 2×), not layout breakpoints; they are independent of
the canonical breakpoints in [03-layout](./03-layout.md#specifications--parameters), which
govern grid changes, not image selection. Width descriptors REQUIRE a `sizes`
attribute (they don't work without it — web.dev). Cap retina delivery at 2× DPR: "the human
eye is unable to benefit from a DPR of 3" (web.dev) — never ship 3× variants.
LCP hero: `fetchpriority="high"`, never `loading="lazy"`; everything below the fold:
`loading="lazy" decoding="async"`. Always set `width`/`height` attributes (CLS).

**Color profiles:**

- Default: convert ALL UI and content imagery to **sRGB (IEC 61966-2.1)**. Color-managed
  browsers treat untagged images as sRGB, so sRGB assets may ship untagged (smaller files)
  — but never ship a P3 image with its profile stripped (it will render oversaturated).
- Wide-gamut: Display P3 covers ≈ 25–50% more chromaticities than sRGB (WebKit). Use a
  P3-tagged variant ONLY for saturated hero art, served via
  `<source media="(color-gamut: p3)">`; keep the sRGB variant as default.
- Phone photos arrive tagged Display P3 — convert to sRGB unless the P3 path above is used.
- Treat all AI-generator output as sRGB unless the tool documents otherwise.
- P3 CSS color tokens / OKLCH architecture: owned by
  [05-color](./05-color.md#specifications--parameters) — do not define tokens here.

### 4.5 WebGL texture, normal-map & matcap creation (creation only — compression/KTX2: doc 09)

| Asset | Creation spec |
|---|---|
| Tiling texture | Power-of-two: 512/1024/2048 (2048 max for web). Prompt: "seamless tileable texture, top-down, even diffuse lighting, no shadows". VERIFY tiling by offsetting 50%/50% and inspecting the cross-seam; AI output usually fails — fix seams manually or use Poly Haven CC0 photoscanned PBR sets (≥ 8K source, downscale) |
| Height/displacement | From luminance of albedo (Materialize, free/open source: diffuse → height → normal/AO/smoothness) or generated grayscale; spec per §4.3 displacement row |
| Normal map | Generate from height map (Materialize or equivalent); strength 1.0–2.5 for subtle web materials. **OpenGL convention (Y+/green-up)** for three.js; for DirectX-style sources invert green or set `material.normalScale.y = -1`. Export 8-bit PNG RGB (16-bit master if banding), NO sRGB profile — normals are non-color data |
| Matcap | 512×512 PNG sRGB (1024 only for full-screen close-ups); shaded sphere inscribed touching all four edges; bake one key light + rim into the sphere render. Sources: nidorx/matcaps library (24-bit PNG, sRGB, grouped by resolution — check repo license before commercial use: UNVERIFIED — confirm before use) or render your own in Blender (Vianvolaeus blender-matcap-generator) with brand palette tokens |
| AO/roughness | From Materialize height pipeline; 8-bit grayscale, channel-pack later in the doc-09 pipeline |

### 4.6 IP / licensing & brand-consistency risk register

| Risk | Fact (verified 2026-06-12) | Mitigation |
|---|---|---|
| No copyright in AI output | USCO Part 2 report (2025-01-29): purely AI-generated outputs are not copyrightable; prompts alone ≠ authorship; human contributions protectable case-by-case | For ownership-critical art use human authorship; document human selection/arrangement/edits; disclose AI content in US registrations |
| Training-data litigation | Getty v Stability AI (UK High Court, 2025-11-04): model weights held not to be infringing "copies"; limited historic TRADEMARK finding for watermark-bearing outputs; UK training legality left undecided | Ban "in the style of [living artist/named library]" prompts; reject any output with watermark-like artifacts |
| Unindemnified commercial use | Adobe Firefly: trained on Adobe Stock/licensed/public-domain content; IP indemnification on paid business/enterprise plans — competitors don't offer equivalents | Route risk-averse clients to Firefly or CC0 (Poly Haven) assets |
| Model license traps | FLUX.2 family: Klein = Apache 2.0; **Dev = non-commercial** source-available; Pro/Flex API-only. BRIA RMBG-2.0 background remover: commercial use requires a Bria agreement (UNVERIFIED — confirm before use) | Check the specific weight license, not the family name, before self-hosting |
| Provenance watermarks | Nano Banana Pro output always carries invisible SynthID; visible watermark removed only on Ultra tier | Budget the right tier; keep provenance metadata intact |
| Style drift / silent model updates | Midjourney default model changed V7 → V8.1 on 2026-06-10/11; hosted models update without notice | Pin model versions in the generation log; re-validate the style anchor after any model change |
| Trademark/likeness hallucination | Generators reproduce logos, celebrity likenesses, branded products | Manual review pass on every shipped asset at 100% zoom; `--no logo, watermark, text` by default |

**Generation log (mandatory, one JSON per asset):**
`{ "tool": "", "model_version": "", "seed": 0, "prompt": "", "negative": "", "refs": [], "style_anchor": "", "date": "", "license_tier": "", "human_edits": "" }`

## Code Examples

**1. Responsive hero with AVIF → WebP → JPEG and P3 art direction:**

```html
<picture>
  <!-- P3-tagged variant only for wide-gamut screens -->
  <source media="(color-gamut: p3)" type="image/avif"
          srcset="hero-p3-1280.avif 1280w, hero-p3-1920.avif 1920w, hero-p3-2560.avif 2560w"
          sizes="100vw">
  <source type="image/avif"
          srcset="hero-960.avif 960w, hero-1280.avif 1280w, hero-1920.avif 1920w, hero-2560.avif 2560w"
          sizes="100vw">
  <source type="image/webp"
          srcset="hero-960.webp 960w, hero-1280.webp 1280w, hero-1920.webp 1920w, hero-2560.webp 2560w"
          sizes="100vw">
  <!-- width/height prevent CLS; LCP image: fetchpriority=high, NEVER loading=lazy -->
  <img src="hero-1920.jpg" width="1920" height="1080" alt="…"
       fetchpriority="high" decoding="async">
</picture>
```

**2. SVG duotone filter (shadow/highlight stops = 05-color tokens):**

```html
<svg width="0" height="0" aria-hidden="true">
  <filter id="duotone" color-interpolation-filters="sRGB">
    <!-- Rec.709 luma desaturation -->
    <feColorMatrix type="matrix" values="0.2126 0.7152 0.0722 0 0
                                         0.2126 0.7152 0.0722 0 0
                                         0.2126 0.7152 0.0722 0 0
                                         0 0 0 1 0"/>
    <!-- tableValues = [shadow, highlight] per channel; fill from color tokens -->
    <feComponentTransfer>
      <feFuncR type="table" tableValues="0.07 0.98"/>
      <feFuncG type="table" tableValues="0.10 0.93"/>
      <feFuncB type="table" tableValues="0.25 0.86"/>
    </feComponentTransfer>
  </filter>
</svg>
<style>.duotone { filter: url(#duotone); }</style>
```

**3. Film grain overlay (static by default; animation is opt-in + reduced-motion safe):**

```html
<svg width="0" height="0" aria-hidden="true">
  <filter id="grain">
    <!-- baseFrequency 0.65–0.9 = grain size; numOctaves 2 = detail; stitchTiles avoids seams -->
    <feTurbulence type="fractalNoise" baseFrequency="0.8" numOctaves="2" stitchTiles="stitch"/>
    <feColorMatrix type="saturate" values="0"/>
  </filter>
</svg>
<style>
  .grain { position: fixed; inset: 0; pointer-events: none;
           filter: url(#grain); opacity: 0.06; mix-blend-mode: overlay; }
  @media (prefers-reduced-motion: no-preference) {
    .grain--animated { animation: grain-shift 0.9s steps(8) infinite; } /* ~9 fps film look */
  }
  @keyframes grain-shift { to { transform: translate(2%, -2%); } }
  /* canonical reduced-motion pattern: ./07-animation-choreography.md */
</style>
```

**4. Matcap material preview (three@0.184.0 per _facts.md):**

```js
import * as THREE from "three"; // three@0.184.0 — npm install three

const matcap = new THREE.TextureLoader().load("/textures/matcap-512.png");
matcap.colorSpace = THREE.SRGBColorSpace;            // matcaps are color data → sRGB

```

## Mode-Specific Guidance

### Create from scratch
1. Write the art-direction brief (§4.1) from the archetype chosen in
   [01-visual-motion](./01-visual-motion.md#specifications--parameters).
2. Moodboard: 12–20 candidates across 2–3 tools; cull to 3–5; promote 1 style anchor.
3. Freeze the consistency kit (model+version, seed, sref/refs, style suffix) and produce the
   full output matrix; log every generation (§4.6).
4. Run treatments and exports per §4.3/§4.4; hand textures to the doc-09 pipeline.

### Re-create from existing site (reverse-engineering)
1. Inventory the live site's imagery: slots, aspect ratios, formats (DevTools → img/network),
   treatments (grain? duotone? halftone pitch/angle?), and palette mapping.
2. Reverse the treatment recipe into §4.3 parameters; sample duotone stop colors and match to
   the recreated token set in [05-color](./05-color.md#specifications--parameters).
3. Generate NEW assets matching the system's style — never copy/redistribute the originals
   (photography and illustration remain copyrighted even when the AI output around them isn't).
4. Document inferred brief + consistency kit so future assets stay coherent.

### Modify an existing system
1. Load the existing generation log and style anchor; reuse the original consistency kit.
   If the original model version is retired, regenerate the style anchor first and diff it
   against the old anchor before producing new assets.
2. New assets must pass a side-by-side audit against 3 existing system assets (palette,
   lighting direction, grain level, treatment) before shipping.
3. Append to the generation log; never overwrite prior entries (provenance history).

## Quality Checklist

- [ ] Art-direction brief exists and every asset's log entry references it.
- [ ] Model + version pinned; seed/sref/refs recorded per asset (log JSON complete).
- [ ] Every asset generated native to its slot ratio (§4.2 table) — no destructive crops.
- [ ] Master ≥ 2× largest shipped width; lossless master archived.
- [ ] AVIF + WebP + JPEG fallback present; per-slot byte budgets met (§4.4).
- [ ] `sizes` present wherever width descriptors are used; no 3× DPR variants.
- [ ] LCP image has `fetchpriority="high"` and is not lazy-loaded; all images have width/height.
- [ ] All shipped imagery sRGB; any P3 variant is ICC-tagged and gated by `(color-gamut: p3)`.
- [ ] Treatments use exact §4.3 parameters; animated grain has reduced-motion fallback.
- [ ] Tiling textures pass the 50%-offset seam test; normal maps are Y+ and untagged (linear).
- [ ] No watermarks, logos, readable gibberish text, or celebrity likeness at 100% zoom.
- [ ] Licensing tier verified per tool (FLUX.2 Dev ≠ commercial; Firefly for indemnified work).

## Anti-Patterns

1. **Prompting without a brief** — produces a mood-board site, not a design system.
2. **Cropping one master into every ratio** — composition collapses; generate per-ratio.
3. **Chained upscaling** (2× then 2× again) — plastic texture, ringing halos; regenerate instead.
4. **Generative upscalers on faces/products** — hallucinated detail = brand-damaging fakery.
5. **Shipping PNG photos** — 3–10× the bytes of AVIF for zero visible gain; PNG is for masters
   and hard-edged alpha only.
6. **Stripping the ICC profile from P3 exports** — neon oversaturation on wide-gamut displays.
7. **Lazy-loading the LCP hero** — instant Core Web Vitals failure (budget anchors: doc 09).
8. **Uniform AI gloss** ("the AI look": centered subject, teal-orange, smooth skin, volumetric
   haze) — exactly what award juries reject; enforce the negative list and treatments.
9. **"In the style of [living artist]" prompts** — legal exposure + derivative aesthetics.
10. **No generation log** — irreproducible assets; `modify` mode becomes guesswork and USCO
    disclosure becomes impossible.
11. **Baked-in text inside generated images** — unlocalizable, inaccessible, usually misspelled;
    set type in HTML/SVG per [04-typography](./04-typography.md#specifications--parameters).
12. **Animated grain with no reduced-motion path** — accessibility failure (pattern owner: doc 07).
